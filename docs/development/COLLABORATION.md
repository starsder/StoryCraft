# StoryCraft 项目协作说明

本文档用于规范 StoryCraft 团队内部的仓库协作流程。所有贡献者请在参与开发前阅读并遵循本说明。

## 1. 分支模型

本项目采用简化的 Git 分支模型：

- `main`：稳定分支，仅用于发布已通过验证的版本。禁止直接在该分支上提交代码或修改。
- `develop`：开发分支，日常开发与功能集成在此进行。所有常规开发工作都应从该分支切出并合并回该分支。
- `feature/<功能名>`：功能分支，用于开发单个新功能或修复。从 `develop` 切出，完成后通过 Pull Request 合并回 `develop`。
- `hotfix/<问题名>`：紧急修复分支，用于修复 `main` 线上的严重问题。从 `main` 切出，修复后同时合并回 `main` 和 `develop`。
- `release/<版本号>`：发布分支（按需使用），从 `develop` 切出，用于版本发布前的最终验证与调整。

## 2. 日常开发流程

### 2.1 新功能开发

1. 确保本地 `develop` 分支为最新：
   ```bash
   git checkout develop
   git pull origin develop
   ```
2. 从 `develop` 切出功能分支，命名规范为 `feature/<功能名>`，例如 `feature/story-outline`：
   ```bash
   git checkout -b feature/story-outline
   ```
3. 在功能分支上进行开发，保持提交粒度合理、语义清晰。
4. 开发完成后，将功能分支推送到远程仓库，并发起 Pull Request 请求合并到 `develop`。
5. 至少需要一名 reviewer 审批通过后，方可合并。

### 2.2 紧急修复

1. 从 `main` 切出 `hotfix/<问题名>` 分支。
2. 修复完成后发起 Pull Request，同时合并回 `main` 和 `develop`。
3. 修复完成后删除该 hotfix 分支。

### 2.3 禁止事项

- 禁止直接在 `main` 分支上修改或提交代码。
- 禁止将未通过 review 的代码合并到 `develop` 或 `main`。
- 禁止在功能分支中开发多个不相关的功能。
- 禁止提交包含敏感信息（API Key、密码等）的代码。

## 3. 提交规范

### 3.1 Commit Message 格式

采用基本的中文描述格式，结构清晰即可：

```
<类型>: <简短描述>

<详细说明（可选）>
```

常见类型：

- `feat`：新功能
- `fix`：修复问题
- `docs`：文档更新
- `refactor`：重构代码
- `test`：测试相关
- `chore`：构建、工具、依赖等杂项

示例：

```
feat: 增加故事大纲生成器

支持基于角色设定和世界观自动生成三级故事大纲。
```

### 3.2 提交粒度

- 每个提交应只做一件事，便于回滚和 review。
- 避免一次性提交大量不相关的改动。
- 提交前确保代码可以运行，不引入明显错误。

## 4. 分支命名规范

| 分支类型 | 命名示例 |
|---|---|
| 功能分支 | `feature/story-outline` |
| 修复分支 | `feature/fix-plot-gap` |
| 文档分支 | `docs/collaboration-guide` |
| 紧急修复 | `hotfix/api-key-leak` |
| 发布分支 | `release/v0.1.0` |

命名使用小写字母、数字和连字符 `-`，避免使用下划线和空格。

## 5. Pull Request 规范

### 5.1 PR 标题

PR 标题应简洁说明本次变更：

```
[feat] 增加故事大纲生成器
[fix] 修复角色关系重复生成的问题
[docs] 更新协作说明
```

### 5.2 PR 描述

PR 描述应包含以下内容：

- 本次变更的目的
- 主要改动点
- 相关的 Issue 编号（如有）
- 测试方式或验证结果

### 5.3 合并条件

- 代码通过 review
- 至少一名团队成员批准
- 没有冲突
- 相关测试或基础检查通过（后续接入 CI 时适用）

### 5.4 涉及依赖或技术栈的变更

若 PR 涉及新增运行时依赖、升级版本或调整技术方案，除常规说明外，还需满足：

- 说明新增依赖的明确理由和替代方案。
- 更新 `pyproject.toml` 中的依赖声明。
- 同步更新 `docs/development/TECH_STACK.md`。
- 确保新增依赖不引入版本冲突或安全/许可证问题。

本项目遵循**最小依赖原则**：框架级运行时程序当前仅依赖 `langgraph`。新增依赖需经过团队讨论。

## 6. Issue 使用规范

- 新功能、Bug、设计讨论应通过 Issue 记录。
- Issue 标题应清晰概括问题。
- 描述中说明背景、期望结果和实际结果。
- 认领 Issue 时在评论中说明，避免多人重复工作。

## 7. 本地开发环境

### 7.1 环境初始化

使用 `pyproject.toml` 安装依赖：

```bash
# 仅安装运行依赖
pip install -e .

# 开发环境安装（包含测试、静态检查、类型检查工具）
pip install -e ".[dev]"
```

### 7.2 开发约定

- 使用与团队一致的技术栈和版本：Python >= 3.10。
- 在 `configs/` 中存放配置模板，敏感信息不提交到仓库。
- 在 `examples/` 中提供可运行的示例或输出样例。
- 提交前请运行基础检查：格式化、静态检查、类型检查、测试（详见 [代码规范与质量检查](./CODE_QUALITY.md)）。
